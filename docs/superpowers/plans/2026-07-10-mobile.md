# Adaptação Mobile Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Tornar tools.poderoso.io utilizável em telas < 768px (drawer de navegação, painéis empilhados, sem overflow horizontal) mantendo o desktop pixel-idêntico.

**Architecture:** Breakpoint único de 768px via media queries em `app/globals.css`. Como os componentes usam estilo inline (que não expressa media query), as propriedades responsivas migram para classes CSS — exceção já sancionada no arquivo (mesmo caso de `:hover`/`@keyframes`). Único estado novo em JS: `menuOpen` no `ToolsApp` para o drawer.

**Tech Stack:** Next.js 16 (App Router), React 19, TypeScript, CSS puro em `globals.css`, `lucide-react` para o ícone ☰. Sem dependências novas.

## Global Constraints

- UI copy em pt-BR, tom existente (`~/format/json`, labels minúsculas).
- Cores sempre via `var(--color-*)`; nenhum hex novo.
- Classe em `globals.css` só quando inline não expressa (media query) ou 2+ componentes usam.
- Sem dependência nova; sem detecção de viewport em JS.
- Desktop (≥ 768px) sem mudança visual perceptível.
- Repo sem test runner: verificação = `npm run lint`, `npm run build` e inspeção visual a 375px.

**Spec:** `docs/superpowers/specs/2026-07-10-mobile-design.md`

- [ ] **Step 2: Sidebar vira drawer**

`components/layout/Sidebar.tsx` completo:

```tsx
import { NAV_GROUPS, type ToolId } from "@/lib/nav";
import { NavButton } from "./NavButton";

interface SidebarProps {
  active: ToolId;
  onSelect: (id: ToolId) => void;
  open: boolean;
  onClose: () => void;
}

export function Sidebar({ active, onSelect, open, onClose }: SidebarProps) {
  return (
    <>
      {open && <div className="sidebar-backdrop" onClick={onClose} />}
      <nav className={`app-sidebar${open ? " app-sidebar--open" : ""}`}>
        {NAV_GROUPS.map((group) => (
          <div key={group.heading}>
            <div
              style={{
                padding: "14px 16px 5px",
                fontSize: 9.5,
                letterSpacing: "0.18em",
                color: "var(--color-primary)",
                textTransform: "uppercase",
              }}
            >
              {group.heading}
            </div>
            {group.items.map((item) => (
              <NavButton
                key={item.id}
                label={item.label}
                active={active === item.id}
                onClick={() => onSelect(item.id)}
              />
            ))}
          </div>
        ))}
      </nav>
    </>
  );
}
```

(estilos de posicionamento saem do inline para `.app-sidebar`; conteúdo interno inalterado.)

(o `<div>` dos dots ganha `className="header-dots"`; o restante do logo fica igual. No bloco final, o span da tagline ganha classe:)

```tsx
        <span
          className="header-tagline"
          style={{ fontSize: 11, color: "var(--color-line)", letterSpacing: "0.06em" }}
        >
          ferramentas para devs
        </span>
```

Nota: no mobile a media query esconde `.header-dots` e `.header-tagline` (`display: none` da classe vence porque o inline não define `display` nesses casos — o dos dots define `flex`, então a regra mobile precisa de `display: none` na classe, que só se aplica < 768px e o inline `display: flex`... **atenção**: inline vence classe. Por isso o `<div>` dos dots deve MOVER `display: flex` do inline para o CSS:)

Adicionar em `globals.css` (fora da media query, junto de `.menu-btn`):

```css
.header-dots {
  display: flex;
  gap: 7px;
  align-items: center;
}
```

E o JSX dos dots fica sem style de layout:

```tsx
      <div className="header-dots">
        <span style={{ ...DOT, background: "var(--color-danger)" }} />
        <span style={{ ...DOT, background: "var(--color-accent-yellow)" }} />
        <span style={{ ...DOT, background: "var(--color-primary)" }} />
      </div>
```
